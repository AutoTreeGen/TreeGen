# ADR-0069: Invisible-git UX wrapper (save / sync / oops)

- **Status:** Accepted
- **Date:** 2026-05-02
- **Authors:** @autotreegen
- **Tags:** `dev-tooling`, `ux`, `git`

## Контекст

Владелец проекта работает соло и хочет минимизировать когнитивную нагрузку
рутинного git-цикла: правка → ветка → коммит → пуш → PR. Цель — "невидимый
git": одна команда `save "msg"` вместо пяти.

При этом два проектных правила нерушимы:

- CLAUDE.md §5: ❌ прямые коммиты в `main`. Только через PR + ревью.
- ADR-0008: pre-commit hooks никогда не обходим (`--no-verify` запрещён).

Первая итерация задачи предлагала `save = git add -A; commit; push origin main`
плюс warn-only pre-commit и фоновый авто-пуш — что нарушало оба правила
одновременно. Эта ADR фиксирует UX-обёртку, которая сохраняет PR-flow и
ничего не трогает в CI / branch protection.

## Рассмотренные варианты

### Вариант A — Прямой `save → main`

- ✅ Минимум команд для владельца.
- ❌ Нарушает CLAUDE.md §5 (без PR).
- ❌ Делает `--no-verify`-by-default через warn-only pre-commit.
- ❌ Чтобы сделать legitimate, придётся отменять CLAUDE.md §5 и ADR-0008.

### Вариант B — `save → feat/auto-<ts> → PR → CI → auto-merge` (выбран)

- ✅ Сохраняет PR-flow и CI-ворота. Никаких изменений в branch protection.
- ✅ ADR-0008 не затрагивается, pre-commit запускается как обычно.
- ✅ Владелец не пишет `git`/`gh` в повседневной работе.
- ❌ Каждый сейв создаёт PR — небольшой шум в списке PR.
- ❌ Auto-merger должен быть включён, иначе PR-ы накапливаются.

### Вариант C — `save → cherry-pick into rolling PR`

- ✅ Меньше веток.
- ❌ Сложнее в реализации (нужно отслеживать "текущий накопитель").
- ❌ Хуже rollback semantics: один откат тянет несколько изменений.

## Решение

Выбран **Вариант B**. Обёртка лежит в `scripts/dev-aliases.ps1` и
загружается через `$PROFILE`. Три команды:

- `save <msg>` — создаёт ветку `feat/auto-<timestamp>`, стейджит всё
  (`git add -A`), коммитит, пушит, открывает PR через `gh pr create`,
  возвращает на `main`.
- `sync` — `git pull --ff-only origin main`.
- `oops` — мягкий откат локального непушеного коммита через
  `git reset --soft HEAD~1` с подтверждением; либо подсказка
  `gh pr close <num>`, если коммит уже на origin.

Force-push, `--no-verify`, `--admin merge`, изменения CI / branch protection
обёрткой не используются. Auto-merge закрывает PR-ы при зелёном CI
(см. `scripts/auto-merge.ps1`).

## Последствия

- Положительные:
  - Владелец работает в режиме "редактор → save msg" без явного git/gh.
  - PR-flow и CLAUDE.md §5 сохранены: каждое изменение проходит CI и review.
  - ADR-0008 не нарушается: pre-commit запускается как обычно, без bypass.
- Отрицательные / стоимость:
  - Списки PR в GitHub Activity забиваются `feat/auto-*` ветками.
  - `git add -A` стейджит всё untracked. Это требует, чтобы untracked-debris
    (личные дизайн-материалы, `*.bak`, локальные скрипты) был покрыт
    `.gitignore`. Большинство дизайн-паттернов уже добавлены через PR #167;
    эта PR дополняет недостающие: `*.bak` (бэкап-копии редактора) и
    `/saved-patches/` (локальная папка с патчами от агентов).
- Риски:
  - Если auto-merger выключен или зависнет — PR-очередь забьётся. Owner
    может вмешаться через `gh pr list --author @me` и закрыть/смерджить вручную.
  - `oops` для непушеных коммитов делает `git reset --soft` (НЕ `--hard`),
    рабочий каталог сохраняется. Для запушеных коммитов wrapper не делает
    force-push — откат через закрытие PR.

## Когда пересмотреть

- Если PR-шум станет проблемой (>20 авто-PR/день) — добавить группировку
  или squash-then-PR режим.
- Если владелец вернётся к ручному git-циклу для крупного фичи — обёртка
  становится опциональной, не удалять.
- Если auto-merger перестанет ловить эти PR-ы — пересмотреть
  `scripts/auto-merge.ps1`.
