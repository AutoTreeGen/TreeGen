# ADR-0002: Структура монорепо

- **Status:** Accepted
- **Date:** 2026-04-25
- **Authors:** @vladimir
- **Tags:** `architecture`, `tooling`, `foundation`

## Контекст

Проект состоит из нескольких независимых, но взаимосвязанных компонентов:
переиспользуемые Python-пакеты (парсер GEDCOM, ДНК-анализ, движок гипотез),
несколько FastAPI-сервисов, Next.js-фронтенд, Terraform-инфраструктура.

Нужно решить, держать всё в одном репозитории (monorepo) или разнести по
нескольким (polyrepo). Развивается одним человеком на старте, потенциально
небольшой командой позже.

## Рассмотренные варианты

### Вариант A — Monorepo (один git-репозиторий)

- ✅ Атомарные изменения через слои (схема БД ↔ парсер ↔ API ↔ фронт).
- ✅ Единый CI, единый набор инструментов, единое версионирование.
- ✅ Простая навигация и кросс-ссылки в IDE.
- ✅ Хорошо работает с Claude Code (один CLAUDE.md, один контекст).
- ❌ Растущий размер репо со временем.
- ❌ Нужна дисциплина по разделению ответственности.

### Вариант B — Polyrepo (репо на каждый компонент)

- ✅ Чёткая граница ответственности.
- ✅ Независимые релизные циклы.
- ❌ Сложнее координировать изменения, идущие через несколько слоёв.
- ❌ Дублирование CI / tooling / dev-окружения.
- ❌ Хуже для соло-разработчика на ранней стадии.

## Решение

Принят **Вариант A — Monorepo** со следующей структурой:

```
autotreegen/
├── packages/         # переиспользуемые Python-пакеты (uv workspace)
├── services/         # FastAPI-сервисы (uv workspace)
├── apps/web/         # Next.js фронтенд (pnpm workspace)
├── infrastructure/   # Terraform / k8s / Alembic
├── docs/             # архитектура, ADR, спеки
└── scripts/          # вспомогательные скрипты
```

Менеджеры пакетов:
- Python: **uv** (workspace через `[tool.uv.workspace]` в корневом `pyproject.toml`).
- JS/TS: **pnpm** (workspace через `pnpm-workspace.yaml`).

CI: один GitHub Actions workflow с параллельными jobs (Python / Frontend / pre-commit).

## Последствия

**Положительные:**
- Изменение, проходящее через парсер → миграцию → API → UI делается одним PR.
- Единая dev-среда (`docker compose up` + `uv sync` + `pnpm install`).
- Лёгкий onboarding (одна точка входа — `README.md`).

**Отрицательные:**
- При росте кодовой базы CI может стать медленным — придётся внедрять
  selective testing (например, через `nx`, `turbo` или собственный таргетинг).
- Нужна дисциплина: не превращать `packages/shared-models` в свалку.

**Риски:**
- Если позже потребуется полностью отделить какой-то компонент (например,
  `packages/gedcom-parser` как open-source), миграция возможна, но потребует
  чистки git-истории.

## Когда пересмотреть

- Размер репо > 5 GB.
- CI > 30 минут на полный прогон.
- Команда > 10 человек.
- Появляется потребность в open-sourcing отдельных пакетов.
